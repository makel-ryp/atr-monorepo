## Overview

App Agent ships with two [MCP (Model Context Protocol)](https://modelcontextprotocol.io/) servers, pre-configured for all major AI coding assistants:

| Server | Transport | Purpose |
|--------|-----------|---------|
| **App Agent Docs** | HTTP (`localhost:3000/mcp`) | Documentation, codebase introspection, feature knowledge, runtime logs |
| **Chrome DevTools** | stdio (`chrome-devtools-mcp`) | Browser automation, screenshots, DOM inspection, performance tracing |

Together they give your AI assistant complete context — it can read your docs, explore your code, understand your features, AND see what's happening in the browser.

### Windows Note

The `.mcp.json` (Claude Code) config uses `cmd /c npx` for the Chrome DevTools server on Windows. If you're on macOS or Linux, update it to use `npx` directly:

```json
{
  "command": "npx",
  "args": ["chrome-devtools-mcp@latest"]
}
```

## App Agent Docs Server

Runs at `http://localhost:3000/mcp` whenever the docs are running. Provides **12 tools** across three categories.

### Documentation Tools

#### `list-pages`

Lists all documentation pages with titles, paths, and descriptions.

```
AI: "What documentation is available?"
→ Returns all pages in the docs
```

#### `get-page`

Retrieves the full markdown content of a documentation page.

| Parameter | Type | Description |
|-----------|------|-------------|
| `path` | string | Page path (e.g., `/getting-started`) |

```
AI: "Show me the installation guide"
→ Returns full content of /getting-started/installation
```

### Codebase Introspection Tools

#### `list-apps`

Discovers all applications and demos in the monorepo.

```
AI: "What apps are in this project?"
→ Returns apps and demos with name, packageName, port, description
```

#### `get-app-structure`

Returns the file tree of any app or demo.

| Parameter | Type | Description |
|-----------|------|-------------|
| `appName` | string | App name (e.g., `dashboard`, `my-app`) |

#### `list-components`

Lists all components, composables, pages, layouts, and API routes in an app.

| Parameter | Type | Description |
|-----------|------|-------------|
| `appName` | string | App name |
| `type` | string | Filter: `all`, `components`, `composables`, `pages`, `layouts`, `api` |

#### `get-file`

Reads the source code of any file in the monorepo.

| Parameter | Type | Description |
|-----------|------|-------------|
| `appName` | string | App name or `core` |
| `filePath` | string | Path within the app |

**Security:** Only reads allowed file types (.vue, .ts, .js, .json, .md, etc.) and blocks sensitive paths (node_modules, .env, credentials).

### Feature Knowledge Tools

These tools connect to the [feature knowledge system](/internal/app-agent/ai/llms) — the slug-based knowledge base that documents how the platform works.

#### `explain`

Retrieves knowledge for a feature slug. Returns the full knowledge file or a specific aspect.

| Parameter | Type | Description |
|-----------|------|-------------|
| `slug` | string | Feature identifier (e.g., `i18n-layers`, `runtime-config`) |
| `aspect` | string | Optional: `description`, `overview`, `faq`, `reasoning`, `details`, `history` |

```
AI: "Explain how the layer cascade works"
→ Returns knowledge from core/docs/knowledge/layer-cascade.md
```

#### `record`

Captures knowledge during work sessions. Writes to the knowledge file for a slug.

| Parameter | Type | Description |
|-----------|------|-------------|
| `slug` | string | Feature identifier |
| `aspect` | string | Which section to write: `description`, `overview`, `faq`, `reasoning`, `details`, `history` |
| `content` | string | Markdown content (replaces the existing section) |

#### `introspect`

Returns the runtime profile for a feature slug — registration info, dependency edges, file mappings, and knowledge status.

| Parameter | Type | Description |
|-----------|------|-------------|
| `slug` | string | Feature identifier |

```
AI: "What files implement rate-limiting?"
→ Returns file mappings, edges, invocation counts from the feature registry
```

#### `census`

Returns a coverage report for the feature system — which features are documented, which files are annotated, which knowledge files are orphaned.

| Parameter | Type | Description |
|-----------|------|-------------|
| `scope` | string | Optional: `all`, `untagged`, `undocumented`, `orphaned` |

#### `log-summary`

Aggregate summary of runtime logs — total counts, breakdown by slug and level, recent errors.

| Parameter | Type | Description |
|-----------|------|-------------|
| `since` | string | Optional time window (e.g., `5 minutes`, `1 hour`) |

#### `recent-logs`

Queries recent runtime logs written by `feat.log()` during development.

| Parameter | Type | Description |
|-----------|------|-------------|
| `slug` | string | Optional: filter by feature slug |
| `level` | string | Optional: `log`, `warn`, `error` |
| `since` | string | Optional time window |
| `limit` | number | Max results (default 50) |

## Chrome DevTools Server

Browser automation via the [Chrome DevTools MCP](https://github.com/anthropics/anthropic-tools/tree/main/packages/chrome-devtools-mcp) server. Requires Chrome/Chromium running with remote debugging enabled.

### Capabilities

| Category | What it does |
|----------|-------------|
| **Navigation** | Open pages, navigate back/forward, reload |
| **DOM Inspection** | Take snapshots (a11y tree), read element properties |
| **Interaction** | Click, fill forms, hover, drag, press keys, upload files |
| **Screenshots** | Capture full page or specific elements |
| **Console** | Read console messages, filter by type |
| **Network** | List requests, inspect request/response bodies |
| **Performance** | Start/stop traces, analyze Core Web Vitals |
| **Emulation** | Viewport, dark mode, geolocation, network throttling |

### Example Workflow

**User:** "Check if the dashboard demo loads correctly"

**AI workflow:**
1. `navigate_page` → Opens `http://localhost:3010`
2. `take_snapshot` → Reads the accessibility tree
3. `evaluate_script` → Checks `document.documentElement.lang` and other properties
4. `list_console_messages` → Checks for errors
5. Reports findings

### Starting Chrome with Remote Debugging

The Chrome DevTools MCP server connects to a running Chrome instance. Start Chrome with:

```bash
# macOS
/Applications/Google\ Chrome.app/Contents/MacOS/Google\ Chrome --remote-debugging-port=9222

# Windows
"C:\Program Files\Google\Chrome\Application\chrome.exe" --remote-debugging-port=9222

# Linux
google-chrome --remote-debugging-port=9222
```

## How AI Uses These Tools

### Example: Understanding and modifying a feature

**User:** "Add rate limiting to the dashboard API"

**AI workflow:**
1. `explain("rate-limiting")` → Reads existing knowledge about rate limiting patterns
2. `list-apps` → Finds the dashboard demo
3. `get-app-structure("dashboard")` → Sees the file layout
4. `get-file("dashboard", "server/api/users.ts")` → Reads existing API routes
5. Creates the implementation following established patterns
6. `record("rate-limiting", "details", "...")` → Updates knowledge with what was built

### Why This Works

Traditional AI code generators fail because they have no understanding of project conventions and the context window fills up with irrelevant code. App Agent's MCP approach gives AI systematic exploration of the codebase, discoverable patterns from working demos, and a living knowledge base that grows as the project evolves.

**The AI isn't smarter. The codebase is smarter.**

## Customization

### Adding Custom Tools

Create tools in `core/docs/server/mcp/tools/`:

```ts
// server/mcp/tools/my-tool.ts
import { z } from 'zod'

export default defineMcpTool({
  description: 'What this tool does',
  parameters: z.object({
    input: z.string().describe('Input parameter')
  }),
  handler: async ({ input }) => {
    // Your logic here
    return {
      content: [{ type: 'text', text: result }]
    }
  }
})
```

### Disabling MCP

If needed, disable in `core/docs/nuxt.config.ts`:

```ts
export default defineNuxtConfig({
  mcp: {
    enabled: false
  }
})
```
